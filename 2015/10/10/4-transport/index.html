Netty 传输 (Transport)

本章的内容包含了各种传输(Transport)，以及它们的API和适用场景

  • NIO
  • OIO
  • Local
  • Embedded

基于网络的应用提供了人机沟通的渠道，当然也会从一个地方往另一个地方移动数据，正确的完成移动数据的动作是大部分是依靠网路传输来实现的，但是在网络上传输的东西是一样的，就是字节（Byte over wire）。
传输(Transport)的概念帮助我们远离底层机器级别的数据传送的方式，使用传输的用户仅需知道发送和接受字节就可以完成数据传送。

如果你已经使用过JDK API来编写网络程序，有时候你会发现你需要比预期更多的连接才能完成目的，如果你需要切换阻塞和非阻塞模式，会非常困难，因为针对阻塞和非阻塞模式，JDK提供的API是完全不一样的。

Netty为传输层定义了统一的API接口，使得针对这样的切换非常容易。而且业务代码是和底层传输的实现是分离的，这样就不需要在更改底层传输的时候，花很多精力去重构整个代码，简单来说你可以专注在业务代码的实现上。
在这篇文章里我们将会学习Netty统一的传输API，同时也对比JDK 的传输API. 也会解释各种不同的传输的适用场景，和实现方式。这样你就可以根据你的应用需求选择合适你的选项。理解本章内容的前提是你需要有些基本的Java编程经验，如果有网络编程经验的就更好理解了。
让我们来看一下传输在真实世界里是如何工作的，

4.2 传输API

Netty 传输API的核心是Channel接口,它定义了所有的出站(outbound)动作。下图 显示了Channel的类层次结构：

上图显示了一个Channel都分配一个ChannelPipeline和ChannelConfig。ChannelConfig包含所有Channel的配置信息，并且允许运行时更改这些信息。显然各种传输的实现不同，它们的配置信息也会不同。所以传输会实现ChannelConfig接口的各种实现类，可以通过javadoc查找ChannelConfig的实现类。
ChannelPipeline包含了所有的ChannelHandler的实例，这些ChannelHandler实例来处理出站和入站的数据和事件。这些ChannelHandler的实现处理了应用逻辑，包括数据传输和事件处理。典型的ChannelHandler的用处有：

  • 改变数据传输的格式
  • 异常通知
  • Channel 激活/非活动 active/inactive 通知
  • 从Channel 注册/注销到EventLoop的通知
  • 用户自定义事件通知

4.3 Netty包含的传输

Netty已经自带了一些列的传输，不是所有的传输都是支持协议的，这个意味着你为你的应用选择的传输必须兼容底层的协议，在这里我们会讨论传输和协议之间的关系。
下表显示了Netty提供的传输

名字 包 描述
NIO io.netty.channel.socket.nio 使用java.nio.channels作为底层，使用基于selector的方式进行网络交互
OIO io.netty.channel.socket.oio 使用java.net作为底层，使用阻塞和流的方式进行网络操作
Local io.netty.channel.local 本地传输在同一JVM里作为管道使用
Embedded io.netty.channel.embedded 嵌入式的传输，在非网络环境下使用ChannelHandler。这种方法对于测试ChannelHandler实现很有用

我们来看一下使用最多的NIO传输。

4.3.1 NIO传输–非阻塞的I/O

NIO传输提供了一种完全异步的I/O库，它基于JDK的NIO系统的Selector方式。Selector其实是一种IO多路复用技术(multiplexing)。关于IO多路复用技术的文章很多，这里不再阐述。
Selector的基本概念就是当注册在Selecor上的Channel的状态改变的时候，Selector能接受通知。这些状态包括：

  • 一个新的Channel已经被接受并且就绪
  • 一个Channel的连接已经完成
  • 一个Channel上已经有数据，而且已经准备可以读了。
  • 一个Channel已经可以写入数据了

Netty的应用能响应这些状态的改变，然后Selector被重置可以接受下一个状态变化的通知。这个过程是通过一个线程去检查状态的更改，如有任何变化就相应的分派它们来实现的.
下表的一些值是SelectionKey类定义的，这些值可以单独被使用，也可以几个值被捆绑在一起来定义一系列的状态改变。这些状态改变真是应用渴望接收到的。

名字 描述
OP_ACCEPT 当一个Channel创建，而且一个新连接接受的时候的请求通知
OP_CONNECT 一个连接建立的时候的请求通知
OP_READ 在Channel中的数据准备就绪，可以读时候的通知
OP_WRITE 可以往Channel中写入更多数据的时候的通知，大多数的时候都是可以的。但是当Socket的缓冲已经被写满了的时候，就不会有这个通知，这个通常意味着你写入数据比远端写入的快。

Netty的NIO 传输在内部使用这个模型去接受和发送数据, Netty给用户使用的传输API已经完全隐藏了内部的实现细节。就像前面讲到的一样，这个API对所有的传输都是一样的。下图显示了传输的处理流程：

  1. 新的Channel注册到Selector
  2. Selector处理Channel的状态改变
  3. 先前注册的Channels
  4. Selector.selector()方法阻塞除非新的状态改变被接收到，或者可配置的超时事件发生
  5. 检查是否有新的状态改变
  6. 处理所有的状态改变
  7. 处理其他的任务

上述的处理过程并没有加入多线程线程，eventloop等概念，关于这些细节我们会在单独的文章中分析。

还有个功能只有在Netty NIO 传输中才有，就是所谓的”zero-file-copy”. 它允许你可以快速有效的传送内容。即从文件系统移动到网络堆栈中，而无需想从内核空间拷贝到用户空间。在Netty这个是通过FileRegion.transferTo()来实现的，内部也是通过java.nio.channels.FileChannel.transferTo()来实现的。
然而也不是所有的操作系统都支持这个特性。此外，你不能在一个文件系统是加密的操作系统中使用。但是只要是传输未加工过的数据(raw content)的都是可以的。

4.3.2 OIO 旧的阻塞输入输出(Old Blocking I/O )

在Netty中OIO代表一种妥协，它通过Netty的通用API访问，不是异步的。但又是构建在同步的java.net的基础上。有些拗口。但是仔细想一下，也是能理解。Netty号称是完全异步的、事件驱动的，但是作为一个框架也需要支持同步的场景。所以这就是个妥协，不可否认的是同步的OIO也是有它自己适用的场景的。
假如你需要迁移一些旧的遗留代码到Netty中，但是它使用了一些基础类库是采用阻塞调用的，如JDBC。这个时候把整个逻辑转化为异步的会比较困难。这个时候你可以选择短期内先使用OIO的传输，然后再迁移到完全异步的传输中。
在Java.net的API中，通常有一个线程接受到达ServerSocket的新连接，然后创建一个新的线程在该连接上处理socket的请求。这个是必须的因为对于每个IO操作，java.net API都会阻塞直到动作完成，所以把所有的这些工作由一个线程来完成也是不合适的。一般来讲都会把这些新的请求递交给一个线程池中的一个线程或者一个eventloop组中的一个eventloop来处理。
但是每个动作会阻塞，对于eventloop来讲就会有些麻烦，因为对于eventloop来讲不能确定该动作需要阻塞多久，所以Netty中对于OIO使用了SOCKET的SO_TIMEOUT的属性。关于SO_TIME, JDK文档中是这样描述的“With this option set to a non-zero timeout, a call to accept() for this ServerSocket will block for only this amount of time. If the timeout expires, a java.net.SocketTimeoutException is raised”.就是说设置了SO_TIMEOUT=10那么对于ServerSocket的accept()方法仅仅阻塞10毫秒，如果超过10毫秒，那么将会抛出SocketTimeoutException 异常。对于Netty而言也是这样，在设置了SO_TIMEOUT之后，如果超时发生，Netty捕获这个异常，继续处理eventloop循环。这就是Netty这类异步框架支持OIO传输的唯一方法。这种方法带来的问题就是填写SocketTimeoutException的堆栈开销。下面是OIO的处理流程：

  1. 该线程分配给了这个Socket
  2. Socket连接到远端
  3. 读取数据(可能阻塞)
  4. 读取完成
  5. 处理读取的字节数据
  6. 执行当前socket上提交的其他任务
  7. 尝试再次读取socket上数据

在这里我们讨论了两种Netty最常见的传输，下面我们会讨论其他的传输。

4.3.3 本地传输(同一JVM内通信机制)

Netty 提供了一种本地传输(local transport)，它是专门作为运行在同一Java虚拟机里的客户端和服务端设计的。同样本地传输也是使用和其他传输一样的通用API访问。
在本地传输里，Channel里关联的SocketAddress并没有绑定到实际的物理地址上。而是保存在一个注册表中。只要是服务器在运行它就一直保存在该注册表中，直到Channel关闭了它才被注销。因为本地传输并没有接受实际的网络请求，所以它不能和其他传输通信。

4.3.4 嵌入式的传输(Embedded Transport)

Netty提供了嵌入式的传输，允许你可以嵌入ChannelHander到其他ChannelHanders中，并且把它当做帮助类一样使用.并且添加灵活的操作方式，就像操作你的ChannelHanders一样。这种嵌入的技术通常作为测试ChannelHandler的实现使用，但是也可以实现这样的功能，即作为添加功能到ChannelHandler而无需更改代码的作用。在第十章我们会详细描述如何使用EmbeddedChannel来测试ChannelHandler。

4.4 传输的使用场景

上面我们已经初步的讨论了所有的传输，我们现在来看一下如何选择传输的一些选项。就像前面讲到的一样不是所有的传输都能支持各种协议，这个可能影响你的选择。下表显示了传输和协议支持的矩阵。

传输 TCP UDP SCTP* UDT
NIO x x x x
OIO x x x x

*表示当前仅此次Linux系统。关于如何激活SCTP协议，可以参考各个发行版的Linux的文档。

Ubuntu:
#sudo apt-get install libsctpl

Feora:
#sudo yum install kernel-modules-extra.x86_64 lksctp-tools.x86_64

这里有些场景你可能会遇到：

  • 非阻塞的代码基础(NON-BLOCKING CODE-BASE）
    如果你的代码里没有阻塞的调用或者可以限制这些调用。那么就是一个很好的NIO适用场景。NIO能很好地处理很多并发网络连接的场景，也可以对共享很少的线程就能处理很多连接的场景处理良好。
  • 阻塞的代码
    如果你的代码严重依赖阻塞IO并且你的应用已经经过相应的良好设计。原先也是工作良好，但是由于例如并发用户的增长，系统不能承受这么多并发或者是由于响应变慢而导致用户体验不佳。你可能想把你的代码迁移到Netty非阻塞模式以提升性能。这个时候可以考虑阶段式迁移的方式：即可以先迁移到Netty OIO模式，然后再迁移其他的阻塞代码,最后可以简单的切换到Netty NIO模式。
  • 在同一JVM内通信
    在同一java 虚拟机里通信，而无需导出网络服务是本地传输的最佳使用案例。这个会消除所有的网络开销，但是在应用的代码里又使用相同的Netty API来操作。这个对于业务代码的编写者很有用。既可以使用相同的API来操作各种传输。如果你需要把这些服务导出到网络上，那么仅仅需要简单替换一下，就可以切换到Netty的NIO/OIO模式。
  • 测试ChannelHandler的实现
    如果需要为你的ChannelHandler编写单元测试类，那么可以使用嵌入式的传输。这个将会很容易让你编写测试你的ChannelHandler,而无需创建大量的mock对象(如采用easymock框架). 你的测试类还是仅需使用Netty的通用API,嵌入传输会保证你的ChannelHandler就像和真实的传输一样工作。我们将会在测试的章节中专门讨论它。

下表列出了我们讨论的场景：

应用需求 推荐的传输
非阻塞代码或者准备开始使用Netty的情况,高并发，低延迟，大量网络交互 NIO
阻塞的代码，并发不高，快速开发，网络交互不多 OIO
